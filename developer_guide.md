# Руководство Разработчика VKS-Group (Developer Guide)

Добро пожаловать в кодовую базу VKS-Group! Это руководство создано для быстрого и безболезненного погружения в проект новых разработчиков.

Проект построен на современном стеке **Astro + Tailwind CSS + TypeScript** с жестким фокусом на производительность (MPA-first) и идеальный код.

---

## 🚀 1. Быстрый старт (Quick Start)

### Требования

- **Node.js**: Версия 20.0+
- **Пакетный менеджер**: Быстрый `npm` (хотя можно использовать `pnpm`/`yarn`)

### Запуск проекта локально

1. Склонируйте репозиторий: `git clone https://github.com/.../vks-astro.git`
2. Установите зависимости:
   ```bash
   npm install
   ```
3. Запустите сервер разработки (dev server):
   ```bash
   npm run dev
   ```
   Сайт будет доступен по адресу: http://localhost:4321

---

## 🛠 2. Доступные скрипты `package.json`

Вам доступны следующие команды для работы с кодом:

- `npm run dev` — Запуск локального сервера разработки с горячей перезагрузкой (HMR).
- `npm run build` — Сборка проекта в production-ready статику (в папку `/dist`).
- `npm run preview` — Локальный просмотр сгенерированной production-сборки.
- `npm run astro` — Вызов CLI самого Astro (например, `npx astro sync` для обновления типов TS).

### Стандарты качества (Обязательно перед коммитом)

Мы не пускаем грязный код в ветку `main`.

- `npm run lint` — Запускает строгую проверку ESLint. Ругается на слишком длинные файлы (Anti-Monolith), неиспользуемые переменные и плохие импорты.
- `npm run format` — **ВАЖНАЯ КОМАНДА!** Запускает Prettier. Он не только выровняет отступы, но и _автоматически отсортирует все классы Tailwind CSS_ в правильном порядке. Обязательно запускайте перед `git commit`.

## 🏗 3. Анатомия проекта (Где что лежит?)

Структура `src/` выстроена максимально логично и предсказуемо.

### `src/pages/` (Страницы и Роутинг)

Здесь работает File-based роутинг Astro.

- Страницы: `index.astro`, `uslugi.astro`, `kontakty.astro`.
- Динамические страницы (Услуги/Правовые): `uslugi/[slug].astro`, `[...slug].astro`. Они генерируются функцией `getStaticPaths()` из коллекций контента на этапе сборки.

### `src/content/` (База Данных контента)

Мы используем Content Collections API.

- Весь текст услуг (`it-audit`, `support`) и политик (`privacy`) лежит здесь в формате `.mdx` (Markdown).
- Вы можете менять тексты, писать статьи, добавлять списки прямо в Markdown файлах. На страницах они будут отрисованы с красивым дизайном автоматически благодаря `@tailwindcss/typography` (классы `.prose`).

### `src/components/` (Визуальные кирпичики)

Секретов нет, интерфейс собирается из компонентов:

- `ui/` — Маленькие "глупые" элементы (`Button.astro`, `Icon.astro`).
- `sections/` — Крупные смысловые блоки (`Hero.astro`, `Services.astro`).
- `layout/` — Шапка (`Header.astro`) и Подвал (`Footer.astro`).

### `src/lib/constants.ts` (Глобальный справочник)

**Если вас попросили поменять номер телефона, email или ссылку в меню — вам сюда!** Весь ключевой текст вынесен в константы. Вам не нужно бегать по компонентам и искать хардкод.

## ⚖️ 4. Жесткие правила написания кода (Золотые стандарты)

Проект подвергся масштабному рефакторингу архитектуры. **Любые отклонения от этих правил не принимаются на Code Review:**

### 🚫 Правило 1: НИКАКОГО ХАРДКОДА в Компонентах

Внутри папки `src/components/sections/` **запрещено** хардкодить статические тексты, заголовки и описания прямо в HTML.

- **Как надо:** Передавайте данные через `Astro.props` со Страницы `src/pages/...`, либо запрашивайте из коллекций через `getCollection()`.

### 🚫 Правило 2: Запрет тегов `<img>`

Использование обычного HTML тега `<img>` строго запрещено во избежание CLS (прыжков макета).

- **Как надо:** Импортируйте картинки из `src/assets/` и используйте встроенный компонент `<Image />` из модуля `astro:assets`. Astro сам проставит размеры и сожмет фото. В блоке Hero (_первый экран_) обязательно используйте конструкцию `<Picture formats={['avif', 'webp']} loading="eager" />` для идеального LCP.

### 🚫 Правило 3: Атрибут `alt` обязателен

TypeScript компилятор Astro выдаст ошибку, если вы попытаетесь добавить картинку без `alt="..."`. Доступность (A11y) для слабовидящих — это приоритет.

### 🚫 Правило 4: Запрет кастомного CSS (`<style>`)

Мы пишем на 100% Tailwind CSS. Никаких `<style>` блоков внутри `.astro` файлов (за редким исключением).

- **Как надо:** Используйте utility-классы Tailwind. Для сложных компонентов (кнопки) используйте утилиту `cn()` из `src/utils/cn.ts` (обертка над `clsx` + `tailwind-merge`).

### 🚫 Правило 5: Никаких тяжелых SPA-фреймворков (Zero-JS)

Проект рендерится в HTML. Мы не тащим React или Vue.
Если нужен интерактив (например, мобильное меню) — используйте нативные скрипты `<script>` с обработчиком `DOMContentLoaded` прямо внутри `.astro` файлов. Они будут изолированы и бандлиться самим фреймворком.

## 🚢 5. Деплой (Выкатка на Production)

Сайт собирается в статичное приложение (Static Site Generation - SSG).
Мы используем GitHub Actions (`.github/workflows/deploy.yml`).
Вам достаточно просто сделать `git push origin main`. GitHub сам соберет проект `npm run build` и зальет статичные `HTML/CSS/JS` файлы на GitHub Pages.
SEO-теги, `robots.txt` и `sitemap.xml` генерируются абсолютно автоматически на этапе билда.
